# Plan: Add Cursor Agent Support on New Agent Interface

## Context

The `soph/agent-refactor` branch refactored the agent system from a callback-driven model (agents call framework functions) to a dispatcher-driven model (agents are passive data providers, framework calls agent methods). PR #260 attempted to add Cursor support on the *old* interface. We need to implement Cursor support on the *new* interface.

The new architecture has a single `DispatchLifecycleEvent` dispatcher that routes normalized `Event` objects from any agent. Adding a new agent means:
1. Implement the `Agent` interface (identity + `ParseHookEvent` + transcript methods)
2. Implement `HookSupport` for hook installation
3. Implement `HookHandler` (legacy, for backward compat)
4. Register in `init()` — the framework auto-discovers agents

## Branch

### 3. `cmd/entire/cli/agent/cursor/hooks.go` — Hook installation

Implements `HookSupport` and `HookHandler` interfaces:
- `InstallHooks(localDev, force)` — writes `.cursor/hooks.json`
- `UninstallHooks()` — removes Entire entries from `.cursor/hooks.json`
- `AreHooksInstalled()` — checks for Entire hooks
- `GetSupportedHooks()` — returns supported hook types
- `GetHookNames()` — returns hook verb list (delegates to `HookNames()`)

**Cursor hooks.json format:**
```json
{
  "version": 1,
  "hooks": {
    "sessionStart": [{"command": "entire hooks cursor session-start"}],
    "sessionEnd": [{"command": "entire hooks cursor session-end"}],
    "beforeSubmitPrompt": [{"command": "entire hooks cursor before-submit-prompt"}],
    "stop": [{"command": "entire hooks cursor stop"}],
    "preToolUse": [{"command": "entire hooks cursor pre-task", "matcher": "Task"}],
    "postToolUse": [
      {"command": "entire hooks cursor post-task", "matcher": "Task"},
      {"command": "entire hooks cursor post-todo", "matcher": "TodoWrite"}
    ]
  }
}
```

Note: Cursor's preToolUse/postToolUse support `matcher` field for filtering by tool name (similar to Claude's structure). The `matcher` field is optional.

Pattern to follow: `cmd/entire/cli/agent/claudecode/hooks.go`

## Hook Name → Event Mapping

| Cursor Hook Name | Event Type | Notes |
|------------------|------------|-------|
| `session-start` | `SessionStart` | Same as Claude |
| `before-submit-prompt` | `TurnStart` | Cursor's name for user-prompt-submit |
| `stop` | `TurnEnd` | Same as Claude |
| `session-end` | `SessionEnd` | Same as Claude |
| `pre-task` | `SubagentStart` | Maps from preToolUse[Task] |
| `post-task` | `SubagentEnd` | Maps from postToolUse[Task] |
| `post-todo` | `nil` | Incremental checkpoint, handled directly |

## Key Design Decisions

1. **JSONL transcript format**: Cursor uses the same JSONL format as Claude Code. Reuse the `transcript` package utilities for parsing.

2. **No TranscriptPreparer**: Cursor doesn't need the sentinel-polling that Claude Code uses. The transcript is synchronously written before hooks fire.

3. **conversation_id fallback**: Cursor payloads may send `conversation_id` instead of `session_id`. Parse both, prefer `session_id`.

4. **Matcher support in hooks.json**: Cursor's hooks.json supports an optional `matcher` field on hook entries (similar to Claude's per-tool filtering). Use this for pre-task (Task matcher) and post-todo (TodoWrite matcher).

5. **No settings merge**: Unlike PR #260's `.cursor/entire.json` approach, we use `.entire/settings.json` as the single source of truth. Cursor doesn't need separate settings.
