# Husky v9 Hook Compatibility

## Context

PR #355 fixed hook installation to use `git rev-parse --git-path hooks`, which correctly resolves `core.hooksPath`. When Husky is present, this resolves to `.husky/_/`. However, Husky regenerates `.husky/_/` on every `npm install` (via `prepare: "husky"` in package.json), wiping Entire's hooks.

**The fix**: When Husky is detected, inject Entire's commands into the **user-facing** `.husky/<hookname>` files instead of writing standalone hooks into `.husky/_/`. User-facing files are committed to the repo and survive `npm install`.

Closes #228.

## How Husky v9 Works

```
.husky/
├── _/                  ← generated wrappers (gitignored, regenerated on npm install)
│                         core.hooksPath points here
├── pre-commit          ← user-editable, committed, survives npm install
├── prepare-commit-msg  ← user-editable, committed, survives npm install
└── ...
```

Git calls `.husky/_/<hook>` (wrapper) which delegates to `.husky/<hook>` (user script).

## Changes

### 1. Husky Detection

Add helpers to detect if `core.hooksPath` resolves to a Husky-managed directory:

- `isHuskyHooksPath(hooksDir string) bool` — checks if path ends in `.husky/_`
- `getHuskyUserHooksDir(hooksDir string) string` — returns `.husky/` parent dir (or empty)

Detection: path pattern match (`.husky/_`) + verify `.husky/` directory exists on disk.

### 2. Injection Content

Add `buildHuskyInjectionSpecs(cmdPrefix)` returning hook commands wrapped in markers:

```sh
# BEGIN entire
# Entire CLI hooks
entire hooks git prepare-commit-msg "$1" "$2" 2>/dev/null || true
# END entire
```

No shebang needed — Husky v9 user scripts run in the wrapper's shell context.

### 3. Modified `InstallGitHook()`

Extract existing logic into `installStandardHooks()`. Add branch:

```
InstallGitHook()
├─ GetHooksDir() → resolves core.hooksPath
├─ if isHuskyHooksPath(hooksDir) && .husky/ exists:
│   └─ installHuskyHooks() — inject into .husky/<hookname>
│      └─ also clean up any stale standalone hooks in .husky/_/
└─ else:
    └─ installStandardHooks() — existing behavior
```

`installHuskyHooks()` uses `injectHuskyHookContent()` which:
- Creates new `.husky/<hookname>` if it doesn't exist
- Appends `# BEGIN entire` block to existing files
- Replaces existing block if already present (idempotent)

### 4. Modified `isGitHookInstalledInHooksDir()`

When Husky is detected, check `.husky/<hookname>` files for the `# BEGIN entire` marker instead of checking `.husky/_/<hookname>`.

### 5. Modified `RemoveGitHook()`

Extract existing logic into `removeStandardHooks()`. Add Husky branch that strips `# BEGIN entire` / `# END entire` blocks from `.husky/<hookname>` files. If the file becomes empty after removal, delete it.

### 6. Block Manipulation Helpers

- `extractEntireBlock(content) string` — returns text between BEGIN/END markers (inclusive)
- `replaceEntireBlock(content, newBlock) string` — replaces the marker-wrapped block

### Edge Cases

| Scenario | Behavior |
|----------|----------|
| `entire enable` before `husky init` | `.husky/` doesn't exist, falls back to standard install in `.husky/_/` |
| User removes Husky later | `core.hooksPath` reverts, standard path used; stale blocks in `.husky/` are inert |
| Already has Entire block | `extractEntireBlock` detects it, replaces with current version |
| Stale hooks in `.husky/_/` from PR #355 | `installHuskyHooks()` cleans them up to prevent double execution |
| Worktrees with Husky | `.husky/` is part of the repo, shared across worktrees — works correctly |
