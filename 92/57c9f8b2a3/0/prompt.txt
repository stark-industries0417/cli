# Remove Auto-Commit Strategy

## Context

The auto-commit strategy is no longer needed and should be removed from the codebase. Manual-commit is already the default strategy and provides all necessary functionality. Removing auto-commit will simplify the codebase by:

- Eliminating 2,500+ lines of implementation and test code
- Reducing maintenance burden for a strategy that's not being used
- Simplifying documentation and user guidance
- Removing complexity from the strategy system

Manual-commit is already the default and will remain as the only strategy after this change.

## Migration Strategy for Existing Users

Users who have `"strategy": "auto-commit"` in their settings files need to be automatically migrated to manual-commit. The migration should:

1. **Detect auto-commit settings**: When loading settings, check if `strategy == "auto-commit"`
2. **Print clear warning**: Display a message explaining:
   - Auto-commit strategy has been removed
   - Settings are being automatically updated to use manual-commit
   - Manual-commit provides similar functionality without modifying the active branch
3. **Update settings files**: Automatically change strategy to "manual-commit" and save:
   - `.entire/settings.json` (if it contains the auto-commit setting)
   - `.entire/settings.local.json` (if it contains the override)
4. **One-time migration**: After settings are updated, subsequent runs won't show the warning

**Implementation location**: `cmd/entire/cli/settings/settings.go` - Add migration logic in the `Load()` function after settings are loaded and merged, but before returning.

**Warning message example**:
```
⚠️  Auto-commit strategy has been removed and is no longer supported.
   Your settings have been automatically updated to use 'manual-commit' strategy.

   Manual-commit provides similar session management without creating commits
   on your active branch. See 'entire help' for more details.
```

## Key Decisions

### How to Handle Existing Auto-Commit Users?
**Decision: Automatic migration with one-time warning**

Alternatives considered:
1. ❌ **Fail with error** - Would break existing users' workflows
2. ❌ **Silent fallback** - Would confuse users who wonder why their setting isn't being respected
3. ❌ **Manual migration command** - Adds friction and requires user action
4. ✅ **Automatic migration with warning** - Best user experience, transparent, and one-time

Implementation approach:
- Migration happens in `settings.Load()` function
- Detect `strategy == "auto-commit"` after loading and merging settings
- Print clear warning message to stderr explaining the change
- Update settings file(s) to use "manual-commit"
- Gracefully handle read-only files (log warning but don't fail)
- Migration happens once - subsequent runs won't trigger warning

### Keep or Remove Strategy Selection?
Since there's only one strategy now:
- **Option 1**: Remove strategy selection entirely from `enable` command - simplify to just enable manual-commit
- **Option 2**: Keep strategy infrastructure for potential future strategies
- **Recommendation**: Keep infrastructure (registry.go, Strategy interface) but simplify setup.go to not prompt for strategy selection

### Simplify Debug Command?
- Current: `entire debug auto-commit` subcommand
- After removal: Only manual-commit remains
- **Recommendation**: Keep `entire debug` command but remove strategy-specific subcommands

### Documentation Simplification
- Current: Heavy emphasis on comparing two strategies
- After removal: Focus on how manual-commit works
- **Recommendation**: Rename "Session Strategies" to "Session Management", remove comparison tables
